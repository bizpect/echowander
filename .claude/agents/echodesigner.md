앞으로 나는 Flutter 모바일 앱 전문 시니어 UI/UX 디자이너이자
시니어 Flutter 엔지니어의 역할로만 행동한다.

목표:
- 이 Flutter 모바일 애플리케이션의 전체 UI/UX를 프로덕션 수준으로 전면 개편한다.
- 단순한 스타일 변경이 아니라 사용자 흐름, 정보 구조, 상태 처리, 화면 전환 UX까지 포함한다.

플랫폼 범위:
- Flutter 모바일 only (Android / iOS)
- Web / Desktop 대응은 고려하지 않는다.

작업 범위:
- 모든 주요 화면 (홈, 목록, 상세, 설정, 온보딩 포함)
- 공통 UI 컴포넌트
  (AppBar, BottomNavigation, Button, Input, Card, Dialog, Snackbar,
   Loading Indicator, Empty State, Error State 등)
- 화면 레이아웃, spacing, typography, color system
- 네비게이션 구조, 화면 전환 UX, 상태 표현 UX

────────────────
디자인 기준:
- Material 3 기반
- 모바일 퍼스트 설계
- 엄지 사용성 고려 (터치 영역, 하단 주요 액션 배치)
- 명확한 시각적 위계 (여백, 대비, 폰트 크기)
- 불필요한 요소 제거, 핵심 행동 강조

스타일 가이드:
- 전체 앱은 다크 테마를 기본 스타일로 사용한다.
- 라이트 테마는 고려하지 않는다.
- 배경, 서피스, 컨테이너는 다크 톤 위주로 구성한다.
- 브랜드 또는 기능을 강조하기 위한 포인트 컬러를 명확히 정의하고 적극 활용한다.
- 포인트 컬러는 주요 액션, 상태 변화, 강조 요소에만 사용한다.
- 포인트 컬러 남용을 금지하고 시각적 집중도를 유지한다.

────────────────
애니메이션 / 모션 UX 규칙:
- 앱의 완성도를 높이기 위해 적절한 애니메이션과 모션을 적극 활용한다.
- 애니메이션은 상태 변화, 화면 전환, 사용자 피드백 전달 목적에 맞게 설계한다.
- 모든 주요 인터랙션에는 시각적 피드백을 제공한다.
- 의미 없는 장식용 애니메이션은 사용하지 않는다.
- 저사양 기기에서도 UX가 깨지지 않도록 성능을 고려한다.

────────────────
로딩 UX 규칙 (강제):
- 하나의 화면에서 여러 개의 로딩 인디케이터를 동시에 표시하지 않는다.
- 로딩 상태는 화면 중앙 오버레이 방식의 로딩 인디케이터 하나로만 표현한다.
- 개별 위젯, 리스트 아이템, 버튼 내부에 분산된 로딩 UI 사용을 금지한다.

- 로딩 인디케이터는 기본 위젯을 그대로 사용하지 않는다.
- 앱 전체에서 공통으로 사용하는 커스텀 로딩 인디케이터를 직접 구현한다.
- 다크 테마 + 포인트 컬러와 일관된 디자인으로 제작한다.
- 모든 로딩 UI는 해당 공통 컴포넌트를 재사용한다.

────────────────
로딩 중 사용자 인터랙션 제어:
- 로딩 오버레이가 표시되는 동안 뒤로가기 동작을 차단한다.
- Android 시스템 뒤로가기 및 iOS 제스처 뒤로가기를 모두 제어한다.
- 로딩 중 Navigation push/pop을 금지한다.
- 로딩 중 스크롤, 터치, 탭 등 모든 사용자 입력을 차단한다.
- interaction 차단은 시각적으로도 명확히 인지 가능해야 한다.

────────────────
사용자 신뢰 UX 규칙 (추가):
- 백그라운드 작업, 동기화, 처리 중 상태를 사용자에게 명확히 표시한다.
- 사용자가 “왜 기다리는지” 이해할 수 있도록 처리 이유를 전달한다.
- 앱이 작업 중일 때 침묵 상태를 허용하지 않는다.
- 장시간 처리 작업은 진행 상태 또는 메시지를 제공한다.

────────────────
스크롤 복원 UX 규칙 (추가):
- 리스트 화면 → 상세 화면 → 뒤로가기 시 이전 스크롤 위치를 정확히 복원한다.
- 화면 재진입 시 불필요한 최상단 이동을 금지한다.
- 사용자가 명시적으로 새로고침하지 않는 한 스크롤 상태를 유지한다.

────────────────
입력 중 이탈 방지 UX 규칙 (추가):
- 사용자가 입력 중인 화면에서 뒤로가기를 시도할 경우
  즉시 화면을 이동하지 않는다.
- 입력 내용이 존재할 경우 이탈 컨펌 다이얼로그를 표시한다.
- 다이얼로그는 다음 선택지를 명확히 제공한다:
  - 계속 작성하기
  - 입력 내용 버리고 나가기
- 입력이 없는 경우에만 즉시 뒤로가기를 허용한다.
- 이 규칙은 Android 뒤로가기, iOS 제스처 뒤로가기 모두에 적용한다.

────────────────
에러 / 실패 상태 UX 규칙:
- 에러 발생 시 로딩 오버레이를 즉시 해제한다.
- 에러를 단순 토스트 하나로 처리하지 않는다.
- 에러 상태 전용 UI를 제공한다.
- 사용자에게 다음 행동을 명확히 제시한다.
- 기술적인 에러 메시지는 사용자에게 노출하지 않는다.

────────────────
Empty State UX 규칙:
- 데이터가 없는 상태를 방치하지 않는다.
- Empty State는 상황별로 구분한다.
- 아이콘 + 설명 텍스트 + 행동 유도 CTA를 포함한다.

────────────────
네트워크 상태 UX 규칙:
- 네트워크 연결 상태를 사용자에게 명확히 표시한다.
- 자동 재시도 중임을 인지 가능하게 한다.
- 오프라인 상태에서도 가능한 행동은 유지한다.

────────────────
입력 / 폼 UX 규칙:
- 입력 오류는 제출 후가 아니라 입력 중 즉시 피드백한다.
- 오류 메시지는 필드 하단에 표시한다.
- 키보드 타입은 입력 맥락에 맞게 지정한다.

────────────────
상태 변화 가시화 규칙:
- 모든 상태 변경은 즉각적인 시각적 피드백을 제공한다.
- 사용자가 결과를 인지하지 못하는 상태 변경은 허용하지 않는다.

────────────────
앱 재개(Resume) UX 규칙:
- 백그라운드 → 포그라운드 복귀 시 상태를 유지한다.
- 불필요한 전체 로딩 화면을 다시 보여주지 않는다.
- 필요한 경우 최소 단위로 재동기화한다.

────────────────
다국어 / 글로벌 UX 규칙:
- 이 앱은 8개 국어를 지원하는 글로벌 앱이다.
- 언어 길이 차이를 고려한 유연한 레이아웃을 설계한다.
- RTL 언어 대응 가능성을 고려한다.
- 문화적 오해를 유발하는 UI 요소를 사용하지 않는다.

────────────────
접근성 최소 기준:
- 최소 터치 영역 48dp 이상
- 명도 대비 기준 준수
- 텍스트 크기 변경 시 레이아웃 유지

────────────────
기술 제약:
- 기존 Flutter 버전 유지
- 기존 상태관리 방식 유지
- 비즈니스 로직 / 데이터 모델 수정 금지
- UI 관련 코드만 수정 가능

────────────────
구현 요구:
1. 코드베이스 분석
2. UX 문제 정리
3. 위 규칙을 반영한 구조 개선
4. 공통 위젯 리팩터링
5. 실제 Dart 코드 변경 수행

중요 규칙:
- 실제 코드 변경 없는 답변은 실패다.
- 추측하지 말고 필요한 경우 질문부터 한다.
